title: "Managing subscription lifecycle in Kill Bill"
description: "Create, change, pause, and cancel subscriptions in Kill Bill. Understand bundles, add-ons, billing cycle days, and entitlement blocking states."
---

Kill Bill models customer access to your products through **entitlements**, which combine an access right (entitlement) with a billing record (subscription). Together these form the subscription lifecycle — from creation through plan changes, pauses, and cancellations. This guide walks through each operation and the API calls that drive them.

## Bundles and subscriptions

A **bundle** is a container that groups one base subscription with zero or more add-on subscriptions. All subscriptions in a bundle share the same `bundleId` and, by default, the same billing cycle day (BCD). This structure mirrors common real-world products where a customer buys a base plan and then optionally adds features.

<CardGroup cols={2}>
  <Card title="Bundle" icon="box">
    Groups a base and its add-ons. Identified by `bundleId` and an optional `externalKey`. Cancelling a base subscription cascades to all add-ons in the same bundle.
  </Card>
  <Card title="Subscription" icon="repeat">
    A single billing record tied to one plan. Each subscription tracks its own phase transitions, BCD, and event history.
  </Card>
</CardGroup>

## Creating a subscription

`POST /1.0/kb/subscriptions` creates a new entitlement (and implicitly a new bundle unless you supply a `bundleId`).

You can specify either a `planName` directly, or provide `productName` + `productCategory` + `billingPeriod` + `priceList` and let Kill Bill resolve the plan from the catalog.

```bash
curl -v \
  -u admin:password \
  -H "X-Killbill-ApiKey: bob" \
  -H "X-Killbill-ApiSecret: lazar" \
  -H "Content-Type: application/json" \
  -H "X-Killbill-CreatedBy: demo" \
  -X POST \
  --data-binary '{
    "accountId": "2ad52f53-85ae-408a-9879-32a7e59dd03d",
    "planName": "sports-monthly"
  }' \
  "http://127.0.0.1:8080/1.0/kb/subscriptions"
```

Alternatively, using product/period/priceList resolution:

```bash
curl -v \
  -u admin:password \
  -H "X-Killbill-ApiKey: bob" \
  -H "X-Killbill-ApiSecret: lazar" \
  -H "Content-Type: application/json" \
  -H "X-Killbill-CreatedBy: demo" \
  -X POST \
  --data-binary '{
    "accountId": "2ad52f53-85ae-408a-9879-32a7e59dd03d",
    "productName": "Sports",
    "productCategory": "BASE",
    "billingPeriod": "MONTHLY",
    "priceList": "DEFAULT"
  }' \
  "http://127.0.0.1:8080/1.0/kb/subscriptions"
```

### Date parameters

Three query parameters control when entitlement and billing become effective:

| Parameter | Default | Description |
|---|---|---|
| `entitlementDate` | now | When the customer gains access (service start) |
| `billingDate` | same as `entitlementDate` | When billing starts; can be set in the future for trials |
| `requestedDate` | now | General-purpose effective date; used when neither of the above is set |

```bash
# Start entitlement today, begin billing in 14 days
curl -v \
  -u admin:password \
  -H "X-Killbill-ApiKey: bob" \
  -H "X-Killbill-ApiSecret: lazar" \
  -H "Content-Type: application/json" \
  -H "X-Killbill-CreatedBy: demo" \
  -X POST \
  --data-binary '{
    "accountId": "2ad52f53-85ae-408a-9879-32a7e59dd03d",
    "planName": "standard-monthly"
  }' \
  "http://127.0.0.1:8080/1.0/kb/subscriptions?billingDate=2026-05-07"
```

## Retrieving a subscription

```bash
curl -v \
  -u admin:password \
  -H "X-Killbill-ApiKey: bob" \
  -H "X-Killbill-ApiSecret: lazar" \
  "http://127.0.0.1:8080/1.0/kb/subscriptions/77e23878-8b9d-403b-bf31-93003e125712"
```

## Changing a plan

`PUT /1.0/kb/subscriptions/{subscriptionId}` changes the plan on an existing subscription. You can supply `planName` or product/period/priceList fields.

By default, Kill Bill applies the change policy defined in the catalog rules (e.g., `END_OF_TERM` for downgrades, `IMMEDIATE` for upgrades in trial). You can override the policy with the `billingPolicy` query parameter.

```bash
# Upgrade to super-monthly immediately
curl -v \
  -u admin:password \
  -H "X-Killbill-ApiKey: bob" \
  -H "X-Killbill-ApiSecret: lazar" \
  -H "Content-Type: application/json" \
  -H "X-Killbill-CreatedBy: demo" \
  -X PUT \
  --data-binary '{
    "planName": "super-monthly"
  }' \
  "http://127.0.0.1:8080/1.0/kb/subscriptions/77e23878-8b9d-403b-bf31-93003e125712?billingPolicy=IMMEDIATE"
```

<Tip>
To preview the invoice that would be generated by a plan change before committing it, use the dry-run invoice endpoint. See the [invoicing guide](/guides/invoicing) for details.
</Tip>

### Undoing a pending plan change

If the change is scheduled for the future (end of term), you can reverse it before it takes effect:

```bash
curl -v \
  -u admin:password \
  -H "X-Killbill-ApiKey: bob" \
  -H "X-Killbill-ApiSecret: lazar" \
  -H "X-Killbill-CreatedBy: demo" \
  -X PUT \
  "http://127.0.0.1:8080/1.0/kb/subscriptions/77e23878-8b9d-403b-bf31-93003e125712/undoChangePlan"
```

## Cancelling a subscription

`DELETE /1.0/kb/subscriptions/{subscriptionId}` cancels the subscription. The catalog `cancelPolicy` rules determine whether cancellation is immediate or at end of term. You can override both the entitlement policy and billing policy independently.

```bash
# Cancel at end of current billing period
curl -v \
  -u admin:password \
  -H "X-Killbill-ApiKey: bob" \
  -H "X-Killbill-ApiSecret: lazar" \
  -H "X-Killbill-CreatedBy: demo" \
  -X DELETE \
  "http://127.0.0.1:8080/1.0/kb/subscriptions/77e23878-8b9d-403b-bf31-93003e125712?billingPolicy=END_OF_TERM"
```

```bash
# Cancel immediately — both access and billing stop now
curl -v \
  -u admin:password \
  -H "X-Killbill-ApiKey: bob" \
  -H "X-Killbill-ApiSecret: lazar" \
  -H "X-Killbill-CreatedBy: demo" \
  -X DELETE \
  "http://127.0.0.1:8080/1.0/kb/subscriptions/77e23878-8b9d-403b-bf31-93003e125712?entitlementPolicy=IMMEDIATE&billingPolicy=IMMEDIATE"
```

### Un-cancelling a subscription

If a pending future cancellation was scheduled, you can undo it before it becomes effective:

```bash
curl -v \
  -u admin:password \
  -H "X-Killbill-ApiKey: bob" \
  -H "X-Killbill-ApiSecret: lazar" \
  -H "X-Killbill-CreatedBy: demo" \
  -X PUT \
  "http://127.0.0.1:8080/1.0/kb/subscriptions/77e23878-8b9d-403b-bf31-93003e125712/uncancel"
```

## Billing cycle day (BCD)

The BCD determines the day of the month on which recurring invoices are generated. Kill Bill sets the BCD automatically based on the subscription start date (or the account-level BCD if billing alignment is set to `ACCOUNT`).

You can override the BCD for a subscription at any time:

```bash
curl -v \
  -u admin:password \
  -H "X-Killbill-ApiKey: bob" \
  -H "X-Killbill-ApiSecret: lazar" \
  -H "Content-Type: application/json" \
  -H "X-Killbill-CreatedBy: demo" \
  -X PUT \
  --data-binary '{
    "billCycleDayLocal": 15
  }' \
  "http://127.0.0.1:8080/1.0/kb/subscriptions/77e23878-8b9d-403b-bf31-93003e125712/bcd"
```

<Note>
Changing the BCD affects only future invoices. Any in-flight billing period is prorated if the new BCD falls within it.
</Note>

## Add-on subscriptions

Add-on subscriptions attach to an existing bundle by providing the `bundleId` of the base subscription. The catalog defines which add-ons are `available` for a given base product.

```bash
curl -v \
  -u admin:password \
  -H "X-Killbill-ApiKey: bob" \
  -H "X-Killbill-ApiSecret: lazar" \
  -H "Content-Type: application/json" \
  -H "X-Killbill-CreatedBy: demo" \
  -X POST \
  --data-binary '{
    "accountId": "2ad52f53-85ae-408a-9879-32a7e59dd03d",
    "bundleId": "a8f78db4-9b3e-4b9c-ab3a-001234567890",
    "planName": "oilslick-monthly",
    "productCategory": "ADD_ON"
  }' \
  "http://127.0.0.1:8080/1.0/kb/subscriptions"
```

When the base subscription is cancelled, Kill Bill automatically cancels all add-ons in the same bundle immediately (controlled by the catalog `cancelPolicy` for `ADD_ON` category).

## Creating a base subscription with add-ons in one call

`POST /1.0/kb/subscriptions/createSubscriptionWithAddOns` creates a base and one or more add-ons atomically:

```bash
curl -v \
  -u admin:password \
  -H "X-Killbill-ApiKey: bob" \
  -H "X-Killbill-ApiSecret: lazar" \
  -H "Content-Type: application/json" \
  -H "X-Killbill-CreatedBy: demo" \
  -X POST \
  --data-binary '[
    {
      "accountId": "2ad52f53-85ae-408a-9879-32a7e59dd03d",
      "planName": "sports-monthly",
      "productCategory": "BASE"
    },
    {
      "accountId": "2ad52f53-85ae-408a-9879-32a7e59dd03d",
      "planName": "oilslick-monthly",
      "productCategory": "ADD_ON"
    }
  ]' \
  "http://127.0.0.1:8080/1.0/kb/subscriptions/createSubscriptionWithAddOns"
```

## Blocking states (pause and resume)

Kill Bill uses a **blocking state** mechanism to suspend entitlement, billing, or both without cancelling the subscription. This is used internally by the overdue system and can also be applied manually.

### Block a subscription

`POST /1.0/kb/subscriptions/{subscriptionId}/block`

```bash
curl -v \
  -u admin:password \
  -H "X-Killbill-ApiKey: bob" \
  -H "X-Killbill-ApiSecret: lazar" \
  -H "Content-Type: application/json" \
  -H "X-Killbill-CreatedBy: demo" \
  -X POST \
  --data-binary '{
    "stateName": "STATE_PAUSED",
    "service": "my-service",
    "isBlockChange": false,
    "isBlockEntitlement": true,
    "isBlockBilling": true
  }' \
  "http://127.0.0.1:8080/1.0/kb/subscriptions/77e23878-8b9d-403b-bf31-93003e125712/block"
```

The three boolean flags control which aspects are suspended:

| Flag | Effect |
|---|---|
| `isBlockChange` | Prevents plan changes |
| `isBlockEntitlement` | Revokes customer access |
| `isBlockBilling` | Stops invoice generation |

<Warning>
Blocking states stack. If multiple services apply blocking states, all must be cleared for the subscription to resume normally. Always specify a unique `service` name so you can precisely target your state when clearing it.
</Warning>
